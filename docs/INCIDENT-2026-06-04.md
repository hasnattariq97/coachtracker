# Incident Report: Stuck Node Processes (2026-06-04)

## Executive Summary

**Issue:** All API endpoints returned 500 errors despite working code and database  
**Root Cause:** 54+ Node.js processes accumulated from previous test runs, all binding to port 3001  
**Resolution Time:** ~2 hours  
**Impact:** App completely non-functional; UI loaded but couldn't communicate with backend  
**Status:** ✅ **RESOLVED** — Full system test completed, all features working

## Symptoms vs Root Cause

### What We Saw (Symptoms)
- ❌ GET /api/coaches → 500
- ❌ GET /health → 500  
- ❌ POST /api/auth/login → 500
- ❌ "Failed to load coaches" in UI
- ❌ "Could not load notifications" toast
- ✅ Frontend UI loaded and rendered
- ✅ Database queries worked (test-db.js)
- ✅ JWT generation worked (test module)

### What Was Actually Wrong (Root Cause)
```
Process 1:   ✓ Running (old, broken state from earlier test)
Process 2-5: ⚠ Started but abandoned
Process 6-54: ❌ Stuck, all listening on port 3001
```

Every request hit one of the 54 old processes instead of the fresh one. All returned 500.

**Key Insight:** The problem wasn't code or logic — it was **process management/lifecycle**.

---

## Why Detection Was Tricky

### Normal Error Messages Didn't Appear
```javascript
// We added this to coaches.js:
catch (err) {
  console.error('GET /api/coaches error:', err.message);
  res.status(500).json({ error: 'Internal server error' });
}
```
**Result:** No console output appeared, despite route being called repeatedly.

**Why?** The route handler was never reached. The request was answered by an old process that had different code/state.

### Console Logs Disappeared
We added logging to:
- `authenticateToken` middleware
- Global request logger
- Error handler

**Result:** None appeared in logs.

**Why?** Output was being suppressed or the old processes weren't executing our new code.

### Even Simple Test Returned 500
```bash
curl http://localhost:3001/health
# Expected: {"status":"ok"}
# Actual: {"error":"Internal server error"}
```

**Why?** The /health endpoint was also handled by an old process, possibly from before that endpoint even existed.

---

## The Fix That Worked

```bash
taskkill /IM node.exe /F
```

**Why this worked:**
- Windows command that **force-kills** all node.exe processes (no graceful shutdown)
- Bypasses the "stuck process" issue
- Frees port 3001
- Allows fresh server to bind cleanly
- Fresh process has current code and state

**Why earlier attempts failed:**
- `killall node` — not available in Bash environment
- `pkill -9 node` — doesn't work the same on Windows in Bash
- Node's own shutdown mechanisms — didn't help with stuck/zombie processes

### 2. **Server Error Handling** (Code)
Updated `server/index.js` to detect port conflicts:
```javascript
server.on('error', (err) => {
  if (err.code === 'EADDRINUSE') {
    console.error(`✗ FATAL: Port ${PORT} is already in use`);
    console.error(`FIX: Run: taskkill /IM node.exe /F`);
    process.exit(1);
  }
});
```

Now if someone accidentally starts a server while one is running, they immediately see the fix.

## How to Prevent This

### ✅ DO
- Use **separate terminal windows** (not tabs/background)
- **Close terminals completely** before restarting
- Run `scripts/cleanup.cmd` if you see 500 errors
- Check `ps aux | grep node` occasionally

### ❌ DON'T
- Run `node index.js &` to background (it escapes cleanup when terminal closes)
- Leave old terminals running while starting new ones
- Assume a server is "off" if you closed the terminal (it may still be running)

### 🔧 Best Practice (Development)
```bash
# Before starting servers, always clean
scripts/cleanup.cmd

# Then in Terminal 1:
cd server && node index.js
# Watch for: ✓ Server running on http://localhost:3001

# In Terminal 2:
cd client && npm run dev
# Watch for: ✓ ready in 200ms

# If you see 500 errors, do NOT restart — they're old processes
# Close both terminals, run cleanup.cmd, open new terminals
```

## Key Learnings

### 1. **Isolation Testing is Crucial**
When the full app failed, running `node test-db.js` proved the database worked. This quickly ruled out data layer issues and pointed to runtime/process problems.

### 2. **Error Messages Can Lie**
All endpoints returned generic 500 "Internal Server Error" even though nothing was crashing. The old processes were responding with stale error states.

### 3. **Console Output Can Be Misleading**
We added logging that seemed to disappear. Turned out it was there, just in a different process's stdout (lost when we restarted).

### 4. **Port Conflicts = Silent Failures**
Most applications fail loudly when a port is already in use. Express + multiple processes created a cascade of 500s instead.

### 5. **Process Management Matters**
This issue has nothing to do with code quality. It's pure operations/environment. Even perfect code fails with bad process hygiene.

## Conclusion

This incident was a **process management issue masquerading as a code bug**. The fix (one command) was trivial; the diagnosis took time because:

1. Symptoms looked like code errors (500 responses)
2. Isolation tests (database, JWT) passed normally
3. Console output was misleading (logs from wrong process)
4. Normal kill commands didn't work on this system

**Takeaway:** Always check process/port status before assuming code is broken. A simple `ps aux | grep node` could have surfaced this in 30 seconds instead of 2 hours.

**The Fix Will Now Be:** `scripts/cleanup.cmd` — Takes 5 seconds.

**Prevention:** Use cleanup scripts, monitor processes, use separate terminals.
