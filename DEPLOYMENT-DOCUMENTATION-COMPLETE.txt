================================================================================
  CRITICAL COMMANDS (Copy-Paste)
================================================================================

Deploy code changes:
  railway deployment up --service spectacular-connection -y

Check if deployment succeeded:
  railway deployment list -s spectacular-connection

View live logs (see what's happening):
  railway logs -s spectacular-connection --tail 50

Rollback to previous version (if something broke):
  railway down

Link to project (first time only):
  railway link -p surprising-expression

Set environment variable:
  railway env set KEY=value

Verify service is running:
  curl https://spectacular-connection-production-d07b.up.railway.app/health

================================================================================
  KEY LEARNINGS (PREVENT MISTAKES)
================================================================================

Learning 1: ALWAYS Verify Database Operations
❌ DON'T: console.log('✓ User created')
✅ DO: console.log('✓ User created:', result.rows[0])
→ This catches silent failures immediately

Learning 2: Test ALL Queries After Database Migration
When switching SQLite → PostgreSQL:
  ❌ DON'T: Assume old syntax will work
  ✅ DO: Test every database query with new syntax
  → Saves hours of debugging

Learning 3: Know Your Deployment Platform
Before starting:
  ❌ DON'T: Assume all platforms work the same
  ✅ DO: Check if platform has auto-deploy or manual deploy required
  → Railway requires CLI, Vercel auto-deploys from GitHub

Learning 4: Use CLI for Manual Tasks
When something needs deploying:
  ❌ DON'T: Wait for automation that doesn't exist
  ✅ DO: Use Railway CLI to deploy immediately
  → Takes 1-3 minutes vs. waiting forever

Learning 5: Create Checklists to Avoid Repeating Mistakes
For every deployment:
  ❌ DON'T: Try to remember all steps
  ✅ DO: Use pre-deployment checklist from docs
  → Prevents 90% of issues

================================================================================
  ISSUES WE FIXED (What We Learned)
================================================================================

ISSUE #1: Missing Admin User in Database
Problem: Login endpoint worked but admin@tracker.com didn't exist
Root Cause: DB seeding code logged success without verifying user was created
Fix: Added RETURNING clause, logged actual user object, used ON CONFLICT
Prevention: Always verify database operations return actual data

ISSUE #2: Cron Jobs Using SQLite Syntax in PostgreSQL
Problem: Errors about julianday() and datetime() not existing
Root Cause: Database migrated to PostgreSQL but cron jobs weren't updated
Fix: Converted to PostgreSQL: NOW(), interval math, async/await
Prevention: Test all database queries after migration

ISSUE #3: GitHub NOT Auto-Deployed to Railway
Problem: Code pushed to GitHub, waited for deployment that never happened
Root Cause: I assumed Railway had auto-deploy like Vercel (it doesn't)
Fix: Used railway deployment up CLI to manually deploy
Prevention: Check platform's deployment model before starting

ISSUE #4: Project Not Linked in First Terminal
Problem: Railway CLI said "workspace required in non-interactive mode"
Root Cause: New terminal session, project wasn't linked
Fix: Ran railway link -p project-name
Prevention: Add to pre-deployment checklist

ISSUE #5: /setup Endpoint Returns 404 (First Deploy)
Problem: Endpoint code existed but returned 404 on first deployment
Root Cause: First deployment didn't fully update code
Fix: Did fresh upload with railway deployment up
Prevention: If endpoint 404s after code change, upload code directly (not just redeploy)
